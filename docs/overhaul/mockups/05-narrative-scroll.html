How We Fetch the Latest Videos

When the home page loads, KillrVideo queries the latest_videos table — a dedicated read-optimised table that exists solely to serve this use-case. Cassandra's data modelling mantra is "model for the query, not the schema": because we always want videos sorted by recency within a day, the partition key is the date bucket (yyyymmdd) and the clustering column is added_date DESC, so Cassandra returns rows pre-sorted — no ORDER BY shuffle needed.

Partitioning by day keeps partition sizes bounded. A single LIMIT 10 on a single partition is one of Cassandra's fastest operations.

Design Decisions

The Data Model Behind Video Discovery

Why does latest_videos use yyyymmdd as its partition key instead of simply partitioning by, say, month or user? The answer is partition cardinality. If we partitioned by month, a busy month could accumulate millions of rows in a single partition — Cassandra's sweet spot is tens of thousands of rows per partition. A day bucket keeps each partition at a predictable, manageable size while still enabling efficient chronological queries.

The trade-off: cross-day queries require multiple round trips (one per day). For a "latest videos" feed this is fine — we almost always want today's videos. For analytics spanning months, a different table or a separate analytics pipeline would be appropriate.

Fetching a Single Video

When you open a video page, the frontend sends a single GET request for that video's metadata. The backend runs a single-partition read against videos_by_id, keyed on video_id. Because video_id is the partition key, Cassandra can route the request directly to the correct node — no scatter-gather, no secondary index.

This is Cassandra's strongest access pattern. A single-partition read typically completes in 1–5 ms on co-located infrastructure.

How Ratings Work in Cassandra

Counting ratings in a distributed database is harder than it sounds. A naive approach — read the current count, add one, write back — is a race condition the moment two users submit simultaneously. Cassandra solves this with a dedicated COUNTER data type: an atomic, distributed increment that requires no read-before-write.

The trade-off: counter tables cannot contain non-counter columns except for primary-key columns. That is why video_ratings is a separate table from videos_by_id.

WRITE POST /api/v1/videos/:id/ratings ~4ms src/hooks/useApi.ts:198

Time-Sorted Comments

Comments must appear in reverse-chronological order — newest first. In Cassandra this is achieved without an ORDER BY in the query: the clustering column comment_timestamp DESC means rows are physically stored newest-first on disk. Reading them in order is a sequential disk scan — Cassandra's fastest I/O pattern.

The partition key is video_id — all comments for a video live in the same partition, so a single read fetches them all with no cross-node coordination.
