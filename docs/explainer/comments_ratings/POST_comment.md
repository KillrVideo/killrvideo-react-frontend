# POST /api/v1/videos/{video_id}/comments -- Add Comment to Video

## 1. Overview

This endpoint allows an authenticated viewer to post a comment on a video. Comments
are a core social feature in KillrVideo, enabling viewers to share their thoughts
and engage in discussion beneath any video that has finished processing (status = READY).

When a comment is submitted, the backend performs several operations:

- Validates that the target video exists and is in READY state
- Generates a TimeUUID for the comment identifier (providing both uniqueness and chronological ordering)
- Runs sentiment analysis on the comment text (using a `MockSentimentAnalyzer` that returns a float score)
- Writes the comment to **two** Cassandra tables simultaneously (denormalization)
- Logs user activity by writing to the `user_activity` table

This endpoint demonstrates two fundamental Cassandra data modeling patterns:
**TimeUUID-based ordering** and **denormalization** (writing the same data to
multiple tables optimized for different query patterns).

## 3. Cassandra Concepts Explained

### TimeUUID: A Clock That Doubles as an ID

Imagine you are numbering photographs as you take them: photo-001, photo-002, and so on.
The number tells you both *which* photo it is (identity) and *when* it was taken relative
to others (order). A **TimeUUID** works the same way but at database scale.

A TimeUUID (also called a "Type 1 UUID") encodes:
- A **timestamp** with 100-nanosecond precision
- A **node identifier** (so two servers generating UUIDs at the same instant still get unique values)
- A **clock sequence** (extra uniqueness insurance)

This means that when Cassandra stores comments ordered by their TimeUUID `commentid`,
it is *automatically* ordering them by creation time -- newest first or oldest first,
depending on the clustering order. No separate `created_at` column is needed for sorting.

**Analogy:** Think of TimeUUID as a library book's call number that also encodes the
date the book was acquired. You can shelve books by call number and inherently know
which ones are newer.

### Denormalization: Same Data, Two Homes

In a relational database, you would store comments in a single `comments` table and
use JOINs to answer different questions ("show me comments for this video" vs. "show me
comments by this user"). Cassandra does not support JOINs, so instead we store the
comment in **two tables**, each optimized for a different query pattern:

| Table               | Partition Key | Answers the Question                  |
|---------------------|---------------|---------------------------------------|
| `comments`          | `videoid`     | "What comments are on this video?"    |
| `comments_by_user`  | `userid`      | "What comments has this user posted?" |

This is **denormalization** -- intentionally duplicating data to make reads fast.
The tradeoff is that every write must go to both tables, and the application must
keep them in sync.

**Analogy:** Think of a recipe that appears in a cookbook both under "Italian Food"
(organized by cuisine) and under "Pasta Dishes" (organized by main ingredient).
The recipe text is duplicated, but you can find it quickly no matter which way you
look it up.

### Sentiment Analysis

The backend runs each comment through a `MockSentimentAnalyzer` that returns a
floating-point score between 0.0 (very negative) and 1.0 (very positive). In a
production system, this could be replaced with a real NLP model. The score is stored
alongside the comment for potential future use (content moderation, sorting, filtering).

## 4. Data Model

### Table: `comments` (comments by video)

```sql
CREATE TABLE killrvideo.comments (
    videoid         uuid,
    commentid       timeuuid,
    comment         text,
    userid          uuid,
    sentiment_score float,
    PRIMARY KEY (videoid, commentid)
) WITH CLUSTERING ORDER BY (commentid DESC);
```

**Key design decisions:**

- **Partition key: `videoid`** -- All comments for a single video live on the same
  Cassandra partition. This means "get all comments for video X" is a single-partition
  query (the fastest kind).
- **Clustering column: `commentid` (timeuuid, DESC)** -- Comments within a partition
  are physically sorted newest-first on disk. Fetching the latest comments requires
  no sorting at query time.

### Table: `comments_by_user` (comments by user)

```sql
CREATE TABLE killrvideo.comments_by_user (
    userid          uuid,
    commentid       timeuuid,
    comment         text,
    videoid         uuid,
    sentiment_score float,
    PRIMARY KEY (userid, commentid)
) WITH CLUSTERING ORDER BY (commentid DESC);
```

**Key design decisions:**

- **Partition key: `userid`** -- All comments by a single user are co-located. The
  query "show me everything this user has commented on" is a single-partition read.
- **Same columns, different partition key** -- The data is identical; only the
  primary key structure differs.

### Why Two Tables?

```
Query: "Comments on video X"           Query: "Comments by user Y"
            |                                       |
            v                                       v
    +---------------+                      +-------------------+
    |   comments    |                      | comments_by_user  |
    | PK: videoid   |                      | PK: userid        |
    | CK: commentid |                      | CK: commentid     |
    +---------------+                      +-------------------+
```

Each table is a dedicated "view" of the same data, shaped for a specific access pattern.

## 5. Database Queries

### Backend: `comment_service.add_comment_to_video()`

The backend method performs these steps:

```python
# Pseudocode based on the backend implementation
async def add_comment_to_video(video_id: UUID, user_id: UUID, text: str):
    # Step 1: Validate video exists and is READY
    video = await video_service.get_video(video_id)
    if video is None:
        raise NotFoundException("Video not found")
    if video.status != VideoStatus.READY:
        raise ConflictException("Video is not ready for comments")

    # Step 2: Generate TimeUUID for the comment ID
    comment_id = uuid1()  # TimeUUID -- encodes current timestamp

    # Step 3: Run sentiment analysis
    analyzer = MockSentimentAnalyzer()
    sentiment_score = analyzer.analyze(text)  # Returns float, e.g. 0.72

    # Step 4: Write to BOTH tables (denormalized write)
    comment_doc = {
        "videoid": video_id,
        "commentid": comment_id,
        "userid": user_id,
        "comment": text,
        "sentiment_score": sentiment_score,
    }

    # Insert into comments table (partitioned by video)
    await comments_collection.insert_one(comment_doc)

    # Insert into comments_by_user table (partitioned by user)
    await comments_by_user_collection.insert_one(comment_doc)

    # Step 5: Log activity
    await activity_service.log_activity(user_id, video_id, "comment")

    return comment_doc
```

### Equivalent CQL Statements

```sql

VALUES (

    'Great explanation of Cassandra data modeling!',

    f9e8d7c6-b5a4-3210-9876-543210fedcba,   -- userid
    now(),                                     -- commentid (same timeuuid)
    'Great explanation of Cassandra data modeling!',
```

## 7. Special Notes

### Astra DB / DataStax Considerations

- **Data API vs. CQL:** The KillrVideo backend uses the Astra Data API (JSON/REST)
  rather than raw CQL. The `insert_one()` calls shown above are Data API methods. The
  CQL equivalents are provided for learning purposes.
- **TimeUUID generation:** TimeUUIDs are generated application-side (Python `uuid1()`)
  rather than using Cassandra's `now()` function. This ensures the same TimeUUID is
  used across both table inserts.
- **No BATCH statement:** Although both inserts should logically happen together,
  the backend does not use a Cassandra BATCH. If one insert succeeds and the other
  fails, the tables will be temporarily inconsistent. For a video comments feature,
  this eventual consistency is acceptable.

### Sentiment Analysis

- The `MockSentimentAnalyzer` is a placeholder. It returns a deterministic float
  based on simple text heuristics. In production, you might integrate with a service
  like AWS Comprehend, Google Natural Language API, or a local ML model.
- The `sentiment_score` is stored but not currently used for any filtering or display
  logic in the frontend. It exists for future moderation features.

### Security

- The endpoint requires a valid JWT. The `userid` is extracted from the token -- the
  client cannot impersonate another user.
- Comment text is validated server-side: minimum 1 character, maximum 1000 characters.
- The video must be in READY status. This prevents comments on videos that are still
  being processed or have errored out during upload.

## 8. Developer Tips

### Common Pitfalls

1. **Forgetting to check video status.** If you are building a similar feature, always
   validate that the resource you are commenting on is in a valid state. Allowing
   comments on PENDING or ERROR videos leads to confusing UX.

2. **Inconsistent writes across denormalized tables.** If the second INSERT fails,
   the comment will appear on the video page but not on the user's profile (or vice
   versa). Consider implementing a retry mechanism or a reconciliation job.

3. **TimeUUID vs. random UUID.** Do not use `uuid4()` (random UUID) for comment IDs
   if you need chronological ordering. TimeUUIDs (`uuid1()`) embed a timestamp that
   Cassandra uses for automatic sorting.

### Frontend Integration

The React frontend calls this endpoint via the API client in `src/lib/api.ts`:

```typescript
async addComment(
  videoId: string,
  comment: CommentCreateRequest
): Promise<CommentResponse> {
  return this.request(`/videos/${videoId}/comments`, {
    method: 'POST',
    body: JSON.stringify(comment),
  });
}
```

After a successful comment, the frontend should invalidate the React Query cache for
the video's comment list so the new comment appears immediately:

```typescript
queryClient.invalidateQueries({ queryKey: ['comments', videoId] });
```

### Observing Denormalization

After posting a comment, you can verify the data was written to both tables by calling:

- `GET /api/v1/videos/{video_id}/comments` -- should show the comment
- `GET /api/v1/users/{user_id}/comments` -- should also show the same comment

If one shows the comment and the other does not, you have a denormalization inconsistency.

### Best Practices

- **Keep comment text reasonably short.** The 1000-character limit prevents excessively
  large partition rows. In Cassandra, very large rows (many wide columns or very long
  text values) can cause performance issues during compaction.
- **Monitor partition size.** If a video goes viral and receives millions of comments,
  the `comments` partition for that video could grow very large. Consider implementing
  bucketing (e.g., partition by `videoid` + `comment_month`) for very high-volume
  scenarios.
- **Use optimistic UI updates.** Show the new comment in the UI immediately after the
  POST returns 201, rather than waiting for a full refetch. This gives users instant
  feedback while the cache refreshes in the background.
